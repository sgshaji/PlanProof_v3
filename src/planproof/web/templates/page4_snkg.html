Spatial Normalised Knowledge Graph (SNKG)

Structuring extracted entities into a queryable graph with spatial awareness.

Why a Knowledge Graph?

Extracted entities are flat -- a list of attribute-value pairs with provenance. But planning validation requires relational reasoning: "Is this property within a Conservation Area?" "Do the form address and drawing address match?" "Is the stated site area consistent with the INSPIRE parcel boundary?" These questions need structure, not just data.

The SNKG (Spatial Normalised Knowledge Graph) transforms extracted entities into a property-centric graph where measurements, addresses, zones, and boundaries are connected through typed relationships. This enables multi-hop queries that a flat entity list cannot support.

Example: Conservation Area Query (C006)

Rule C006 asks: "Is the property in a Conservation Area?" If yes, permitted development rights are restricted. The SNKG resolves this by linking the property's postcode to a geocoded location, then checking spatial containment against known Conservation Area boundaries. Without the graph, this would require ad-hoc lookups scattered through the rule logic. 

// Cypher query: find if property is in a Conservation Area
MATCH (p:Property)-[:HAS_ADDRESS]->(a:Address)
MATCH (p)-[:LOCATED_IN]->(z:Zone {type: 'conservation_area'})
RETURN p.id, a.postcode, z.name, z.designation_date

This Cypher query traverses from the property node through its address to check for Conservation Area zone relationships. The SNKG pre-computes these spatial relationships during graph construction so that rule evaluation is a simple graph traversal, not a runtime geospatial operation.

Ablation Evidence: What Happens Without SNKG?

Ablation B removes the SNKG component, forcing the system to work from flat entity lists. The result: the system produces identical verdict accuracy to the full system on the current rule set. This is because the current 8 rules primarily check numeric thresholds rather than relational queries.

However, the SNKG becomes essential for more complex rules that require spatial reasoning -- Conservation Area checks, listed building proximity, flood zone containment. These rules cannot be evaluated from flat entity lists alone.
