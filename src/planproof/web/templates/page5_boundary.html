Boundary Verification

Multi-tier site boundary checking -- the most common reason for application rejection.

Why Boundary Checking Matters

Incorrect site boundaries are the single most common reason for planning application rejection at validation stage. If the red-line boundary on the site plan doesn't match the actual property extent, the entire application is invalid. Planning officers spend significant time cross-referencing site plans against land registry data and OS mapping.

PlanProof implements a three-tier boundary verification system. Each tier provides independent evidence about whether the applicant's stated site boundary is consistent with external data. The tiers are designed to degrade gracefully -- if INSPIRE data is unavailable, the system falls back to visual and scale-bar checks rather than failing entirely.

Tier 1: VLM Visual Alignment

Tier 1

Visual boundary analysis via GPT-4o

The VLM examines the site plan image and assesses whether the red-line boundary appears reasonable for the stated property type and location. It checks for obvious discrepancies: boundaries that include public roads, neighbouring properties, or impossibly large/small areas for the stated address.

Detects: Gross boundary errors, inclusion of public land, missing boundary lines
Limitation: Cannot verify exact boundary position -- visual assessment only

Tier 2: Scale-Bar Measurement

Tier 2

Area comparison using the drawing's own scale bar

If the site plan includes a scale bar, PlanProof measures the boundary area using the stated scale and compares it to the applicant's declared site area. A discrepancy exceeding 15% triggers a flag. This catches cases where the applicant states "400 m2" but the drawing shows a much larger or smaller area.

Threshold: 15% area discrepancy triggers DISCREPANCY_DETECTED
Limitation: Depends on scale bar presence and correct scale notation

Tier 3: INSPIRE Polygon Lookup

Tier 3

Cross-reference against national cadastral data

The INSPIRE dataset contains cadastral parcel boundaries for England and Wales -- over 346,000 parcels in the Birmingham area alone. PlanProof geocodes the application address using postcodes.io, finds nearby INSPIRE parcels, and compares the official parcel area with the applicant's stated site area.

Data source: HM Land Registry INSPIRE polygons (GML format)
Geocoding: postcodes.io free API -- postcode to lat/lng
Limitation: INSPIRE data may not match exact property boundary (leasehold vs freehold)

Combined Boundary Verdict

The three tiers produce independent evidence streams that combine into a single boundary assessment. The logic is deliberately conservative:

Outcome Meaning When Produced
CONSISTENT All available tiers agree the boundary is reasonable At least one tier confirms, none flag discrepancy
DISCREPANCY_DETECTED At least one tier flags a significant mismatch Scale-bar area differs >15%, or INSPIRE area differs significantly
INSUFFICIENT_DATA Not enough evidence to assess the boundary No scale bar, no INSPIRE match, VLM inconclusive

Limitations

Boundary verification in PlanProof detects gross discrepancies only -- it is not survey-grade. A boundary that is off by 1-2 metres will not be flagged. The system is designed to catch the cases that waste the most officer time: boundaries that include neighbouring land, boundaries drawn around the wrong property, or stated areas that are wildly inconsistent with the actual parcel. For precise boundary verification, a land surveyor's assessment remains necessary.
